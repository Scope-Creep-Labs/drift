# Drift — single-server deployment

Everything required to run a complete Drift stack on one Linux box behind a public domain. One Caddy out front terminates TLS via Let's Encrypt and reverse-proxies every internal service.

## What you get

| Service | Purpose | URL |
|---|---|---|
| **drift-frontend** | Drift web UI | `https://<domain>/` |
| **drift-agent** | FastAPI agent + deploy CP + terminal relay | (internal) |
| **drift-postgres** | Drift state | (internal) |
| **vm** | VictoriaMetrics — time-series DB | (internal) |
| **victorialogs** | VictoriaLogs — log DB | (internal) |
| **vmauth** | Auth gateway for remote vmagents | `https://<domain>/vm/` + `/vl/` |
| **vmalert** | Alert rule eval | `https://<domain>/vmalert/` |
| **alertmanager** | Alert routing + silencing | `https://<domain>/am/` |
| **alertmanager-ntfy** | AM webhook → ntfy.sh translator | (internal) |
| **grafana** | Dashboards | `https://<domain>/grafana/` |
| **caddy** | TLS + reverse proxy + basic_auth for vmalert/am | host :80, :443 |
| **reporter-** {vmagent,cadvisor,vector,node-exporter} | Self-scrape of this host into local VM | (internal) |

## Prerequisites

1. A Linux host (any reasonable distro) with Docker + the Compose plugin installed.
2. A public DNS name pointing at the host (A and/or AAAA record). The installer waits to issue TLS until you confirm DNS is in place.
3. Ports **80** and **443** reachable from the public internet (Let's Encrypt's HTTP-01 challenge needs port 80; serving traffic needs 443).
4. ~2 GB RAM minimum; ~10 GB disk for a few months of metric retention.

## Install

```bash
git clone https://github.com/kidproquo/drift.git
cd drift/deploy
./install.sh
```

The installer prompts for:

- **Public domain** + email (Let's Encrypt registration)
- **Drift admin** username + password (bootstrapped on first agent start)
- **vmalert / Alertmanager UI password** — basic-auth credential for `/vmalert/` + `/am/`. Separate from your Drift admin login.
- **LLM model** + matching API key (one of `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY`)
- **ntfy topic** for Alertmanager push notifications
- **B2 / S3** credentials (optional — only needed for Drift Deploy's bundle uploads)
- **Self-scrape labels** (hostname + group used for this host's metrics)

It auto-generates: `DRIFT_SECRET_KEY` (Fernet), `DRIFT_PG_PASSWORD`, the vmauth `reporter` password, and (if you press Enter at the prompts) the Drift admin password + the vmalert/AM UI password. The latter is bcrypt-hashed via the Caddy image. Everything lands in `.env` (mode 600) plus rendered config files under `config/`.

Then `docker compose up -d --build` brings the whole stack up. First start takes a few minutes (image pulls + build).

## Re-running

`install.sh` is idempotent. Re-run it any time to:

- Rotate any prompted secret (Enter at the prompt keeps the current value).
- Change the model + API key.
- Switch ntfy topics.

The auto-generated secrets (`DRIFT_PG_PASSWORD`, `DRIFT_SECRET_KEY`, `REPORTER_PASSWORD`) are preserved across reruns unless you manually clear them in `.env`.

## Day 2 — common operations

```bash
# Watch all logs
docker compose logs -f

# Just one service
docker compose logs -f drift-agent

# Restart after editing .env
docker compose up -d

# Update images (latest drift + VM tags)
docker compose pull && docker compose up -d --build drift-agent drift-frontend

# Reload alertmanager.yml without restarting the container (after the
# agent's LLM tools edit it)
docker compose exec alertmanager wget -qO- --post-data='' http://localhost:9093/am/-/reload

# Stop everything
docker compose down

# Stop + wipe all data (volumes too — destructive)
docker compose down -v
```

## Setting up the first remote device

After install completes, point the public vmauth user (`reporter`) at any remote vmagent by setting `-remoteWrite.url=https://<domain>/vm/api/v1/write` with basic auth `reporter:<REPORTER_PASSWORD from .env>`. For Drift Deploy edge agents, install them with `CP_URL=https://<domain>/api/deploy` — or `https://<domain>/drift/api/deploy` if you chose a subroute layout (PUBLIC_URL ending in `/drift`).

## Troubleshooting

**"TLS issuance failing" / certificate errors**
DNS not yet propagated, or ports 80/443 not reachable. `docker compose logs caddy` shows the ACME error. Once DNS resolves and the host is reachable, Caddy retries every ~minute.

**Drift web UI loads but `/api/*` 502s**
`drift-agent` not healthy. `docker compose logs drift-agent`; common cause is `DRIFT_PG_URL` wrong or Postgres not yet up.

**"credit balance is too low" in chat**
The configured `MODEL`'s API key has no funds. Top up on the vendor's dashboard, or change `MODEL` to a provider you do have credits for.

**Drift agent can't write to `/etc/alerts/drift-managed.yml`**
`config/alerts/` ownership mismatch — install.sh chowns it to uid 999 (the `app` user inside the drift-agent container). If you copied the dir from somewhere else, re-run install.sh or `chown -R 999:999 config/alerts config/alertmanager`.

**Devices on remote networks can't reach this host**
Caddy's TLS issuance needs port 80 + 443 publicly reachable. If your host is behind a NAT, forward both ports or use a tunnel (Tailscale Funnel, Cloudflare Tunnel, etc.) — but note Cloudflare's TLS termination + Caddy's TLS-on-the-other-side adds complexity.
