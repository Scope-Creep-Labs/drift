## v0.1.58

Fix: `list_containers` returned 0 containers for every host because it filtered on the wrong label.

### What changed

`drift-agent/app/tools/metrics.py:226` filtered cAdvisor's `container_last_seen` series by `instance="{host}"`, but for our scrapes `instance` is the scrape endpoint (e.g. `reporter-cadvisor:8080`) — the device name lives on the `host` external_label. Filtering by `host="{host}"` now matches the canonical identifier returned by `list_hosts`, so `list_containers(host=…)` and any follow-up container-scoped queries the agent writes against that hostname actually find data.

The schema description for the `host` parameter was also misleading the model — it called the value an `'instance' label value (e.g. pi:9100)`. Updated to say it's the `host` external_label value (the same identifier `list_hosts` returns), so the model picks the right scope without an extra discovery round-trip.

### Impact

- Any "show me container CPU/memory on `<device>`"-style investigation that started with `list_containers` was returning empty and the agent was bailing. Now works.
- No backend API surface change. Tool input/output schema unchanged on the wire (only the human-readable `description` text moved).

### Operator action

Apply via Software Updates modal. Agent-only — `drift-frontend` image unchanged.

### Migration / breaking changes

None.

### Cache impact

The `list_containers` tool's `description` text in the agent's tools list changed — this invalidates the prompt-cache prefix once across all sessions on first call after the upgrade. Subsequent turns rebuild the cache normally.
