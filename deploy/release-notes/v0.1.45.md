## v0.1.45

Image-only release. Two fixes to v0.1.44's `DRIFT_USER_UID/GID` + `DRIFT_APP_STATE_DIR` flow that only surfaced once a real LSIO-style bundle (Transmission) hit the path.

### Fixes

- **`DRIFT_USER_UID` / `DRIFT_USER_GID` now resolve against the host, not the agent container.** v0.1.44 called `id -u drift` from inside the drift-deploy-agent container, which carries its own `/etc/passwd` and has no `drift` user — so the fallback fired and every bundle saw `PUID=1000` regardless of the host's actual drift UID. Switched to `nsenter -m -t 1 id -u drift`, which enters PID 1's mount namespace (= the host's init, reachable because the agent runs `--pid host`) and reads the host's `/etc/passwd`. `util-linux` is already in the agent image; no new deps.

- **Bind-mount sources under `${DRIFT_APP_STATE_DIR}` are now pre-created and chowned before `docker compose up`.** Docker auto-creates missing host paths for bind mounts as `root:root` (the daemon runs as root and doesn't chown the synthesized dir). An LSIO container running as `PUID=${DRIFT_USER_UID}` then fails to chown its workdirs — surfaces as `[migrations] Permissions could not be set` and downstream `cannot statx '/downloads/complete': No such file or directory`.

  The agent now runs `docker compose config --format json` after generating the override, parses out every `bind`-type volume whose source falls under `$app_state_dir`, and `mkdir -p` + `chown drift:drift` each one before bringing the stack up. Existing host paths are left alone (operator-set perms inside the state dir are preserved).

### Operator action

Apply via the Software Updates modal in the UI (one-click). Image-only — no install.sh re-run. `AGENT_VERSION` bumps to `0.13.1`; each edge device self-updates on its next 30s check-in tick.

After landing, any **new revision** push will run through the corrected code path. Existing apps that were deployed during v0.1.44's broken window (state dir contains files owned by the wrong UID, e.g. the system's `1000`) need a one-time cleanup on the device:

```bash
docker stop <app>
sudo chown -R drift:drift /var/lib/drift-deploy/apps/<app>/state/
# push a new revision to trigger re-apply, OR:
docker start <app>
```

### Migration / breaking changes

None. v0.1.44 bundles that referenced `${DRIFT_USER_UID}` and `${DRIFT_APP_STATE_DIR}` are unchanged at the compose level — only the agent's resolution + pre-create behavior differs. Bundles authored before v0.1.44 don't touch these vars and are unaffected.
