## v0.1.43

Image-only release. Reject app names that Docker Compose's project-name parser would reject — at creation time, before any deployment target ever exists.

### Why

Drift was letting operators create app rows with names that the edge agent later couldn't apply. The canonical case: an app named `NodeRED` (capitals). `docker compose -p NodeRED up` fails with `"invalid project name: must consist only of lowercase alphanumeric, hyphens, and underscores, starting with a letter or digit"`. The CP's retry budget catches it (status → `paused_retries` at 5 attempts) but the app row itself is fundamentally broken from creation — it can never apply on any device. Catching this at `create_app` time saves the operator the diagnostic loop.

### What changed

- **New `validate_app_name()`** in `drift-agent/app/deploy/naming.py`. Regex `^[a-z0-9][a-z0-9_-]{0,127}$` matches Compose's project-name rule. Error message pinpoints which rule was broken (uppercase / leading hyphen-or-underscore / illegal character) instead of just "invalid."
- **REST endpoint** `POST /api/deploy/apps` runs validation; rejects with `400` + the specific message.
- **Chat tools** `create_app` and `fork_app` (target name) run the same check, return `{error: "..."}` on failure so the agent surfaces it cleanly in the chat.
- **AppModal UI** — placeholder now reads `e.g. node-red, reporter, podnot, hello_world`; helper text spells out the full rule including underscores and the start-character constraint; inline `error` state lights up the field red and switches the helper to the rule-violation message as the operator types.

### Operator action

Apply via the Software Updates modal in the UI (one-click). Image-only — no install.sh re-run.

### Migration / breaking changes

None for valid-named apps. Existing apps with invalid names (e.g. uppercase) keep working until you try to create another one with an invalid name. If you want to clean up a legacy bad-name row, delete it in the DB:

```sql
DELETE FROM apps WHERE name = '<the-bad-name>';
```

The `apps` → `app_revisions` and `apps` → `deployment_targets` FKs are `ON DELETE CASCADE`, so revisions and deployment targets follow.
