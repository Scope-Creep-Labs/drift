## v0.1.59

New **device tunnel** feature — operators forward a port on a device's `localhost` to a randomized HTTPS subdomain. Same UX as `ssh -L <local>:localhost:<remote>`, but the destination is a browser tab on a Drift subdomain and there's no SSH key to plumb.

### Why

For "I just want to poke at this app's debug UI on `<device>:3000` for half an hour" — adminer, Grafana on a Jetson, a partner app's status page, etc. Existing options (Tailscale on every device, an SSH jump host) drag a second control plane into the picture; this keeps everything inside Drift's auth + group-access model. The subdomain shape (vs. a path prefix) means proxied apps see their own root paths — anything with hardcoded absolute paths (Grafana is the canary) just works without response rewriting.

### What's new

**Edge:**

- New `edge-agent/tunnel-bridge.py` — sibling to `terminal-bridge.py`, self-updating via SHA comparison on check-in. Multiplexes channels over one persistent WS to the CP; each channel maps to a fresh `localhost:<port>` TCP socket on the device.
- `drift-deploy-agent.sh` now spawns one `tunnel-bridge.py` per `pending_tunnels[]` entry on check-in (paired with a `tunnel_bridge_target_sha` self-update step, same model as terminal-bridge).

**CP:**

- New table `tunnel_sessions` (migration `0015`) — id, device_id, user_id, subdomain_token, port, status, created_at, expires_at, ended_at.
- `POST /api/deploy/devices/{name}/tunnel/open` — mint; returns `{id, url, port, expires_at, ...}`. Auth: cookie + deploy role + group access.
- `GET /api/deploy/devices/{name}/tunnels` — list active tunnels (pending + active). Filtered by group like other deploy GETs.
- `DELETE /api/deploy/tunnels/{id}` — revoke. Only the user who opened it (or an admin) can revoke.
- `GET /api/internal/tunnel/check?domain=...` — Caddy on-demand TLS `ask` hook (no auth — returns 200/404 only).
- `WS /api/deploy/agent/tunnel/ws/{id}` — edge bridge attaches here (bearer = device bootstrap token, cross-checked against the session's device_id).
- New ASGI middleware `TunnelProxyMiddleware` — intercepts requests with `Host: tunnel-<token>.<TUNNEL_BASE_DOMAIN>` BEFORE FastAPI's router, looks up the bridge, multiplexes an HTTP request as a fresh channel, parses the response back via h11 and re-emits as ASGI.
- TTL sweep loop (60s tick) reaps tunnels past `expires_at`.

**Edge image:**

- `drift-agent/Dockerfile` builds `tunnel-bridge.py` into the image; `BUILD_CONTEXT_FILES` (served as the tarball the device pulls at install time) includes it too.

**Frontend:**

- New `TunnelModal.tsx`: port input, **Open tunnel** button, list of active tunnels for the device with copy-URL / open-in-new-tab / revoke. Live-polls every 5s so a freshly attached bridge promotes `pending → active` without manual refresh.
- New `LanIcon` button on each device row in the Investigations sidebar — opens the tunnel modal. Wired through `tunnelUiStore` mirroring `terminalUiStore`.

**nginx (drift-frontend):**

- New server block matching `server_name ~^tunnel-`. Forwards everything to drift-agent on the docker network with `Upgrade`/`Connection` headers (WS-ready) and 1h timeouts. The catchall `_` block continues to serve the SPA + `/api/*` for the base domain.

**Config (`app/config.py`):**

- `TUNNEL_BASE_DOMAIN` (default `""`) — base domain the subdomain pattern matches. Empty = feature disabled (mint returns 503).
- `TUNNEL_DEFAULT_TTL_SECONDS` (default `14400` = 4h).
- `TUNNEL_MAX_CONCURRENT` (default `32`) — CP-wide ceiling.

### Operator action

This release ships the code, but the host needs **two manual prereqs** beyond the usual Software Updates apply:

1. **Wildcard DNS** — add an `A` record:
   ```
   *.dabba.princesamuel.me  →  <CP host IP>
   ```
   (Or whatever your base domain is.)

2. **Caddy** — add a `tunnel-*.<base>` site block + the global `on_demand_tls.ask` allowlist. Snippet at `deploy/Caddyfile.tunnel-example.snippet` in the release tarball — copy into `/etc/caddy/Caddyfile`, substitute the base domain + ask-hook target, `caddy reload`.

3. **CP env** — set in `deploy/.env`:
   ```
   TUNNEL_BASE_DOMAIN=dabba.princesamuel.me
   ```
   The Software Updates apply re-creates the agent container and reads the new value.

Without those, the feature stays dormant (mint returns 503, the modal shows the error, nothing else changes). With them, the modal works end-to-end.

### Migration / breaking changes

Alembic migration `0015_tunnel_sessions.py` adds one table — no destructive change to existing rows. The agent runs migrations on startup, so the Software Updates apply handles it.

### Known limitations (will ship in 0.1.60+)

- **WebSocket forwarding inside the tunnel is not yet supported.** A WS upgrade on the tunneled URL gets a clean 1011 close with reason `tunnel WS forwarding not yet supported`. The proxied app's HTTP surface works; if it falls back to long-polling, that works too. Full WS support needs a wsproto-based frame-translation layer that's larger than this release's scope.
- The agent's per-tunnel WS is rebuilt on a CP restart (in-memory state, like the terminal relay). A short blip while the bridge re-attaches; operators don't need to reopen the modal.

### Cache impact

System prompt and tool registry unchanged. Cache prefix stays byte-stable across the upgrade.
