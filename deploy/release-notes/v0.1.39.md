## v0.1.39

Bundle release. Splits **bundle delivery** from **operational state**: the install.sh now copies what it needs into a canonical state dir (`/var/lib/drift-cp`) and runs every compose operation from there. The bundle directory you extracted the tarball to becomes a pure delivery payload that can be `rm -rf`'d the moment install.sh finishes.

### What changed

- **`DEPLOY_DIR` is fixed at `/var/lib/drift-cp`** (overridable via `DRIFT_STATE_DIR` for hosts where `/var/lib` isn't appropriate). Previously DEPLOY_DIR was wherever you extracted the bundle, which meant `mv` of that directory would silently break the running stack until force-recreate.
- **install.sh writes everything operational under `DEPLOY_DIR`**:
  - `docker-compose.yml` + `docker-compose.external.yml` (copied from bundle)
  - `models.json` (copied from bundle)
  - `.env` (generated; canonical, no separate sidecar)
  - `config/*` (templates rendered from bundle into state)
  - `logs/install-*.log`, `logs/last-answers.env` (sidecar that survives mid-install aborts)
- **`docker-compose.yml` binds the stable absolute path**:
  - `/var/lib/drift-cp:/var/lib/drift-cp:ro` (the install dir mirror)
  - `/var/lib/drift-cp/.env:/etc/drift/cp.env:rw` (the LLM-settings endpoint's write target)
- **All `docker compose` invocations from inside drift-agent** (Software Updates "Apply", LLM Settings "Save") default to `/var/lib/drift-cp`. The legacy `DEPLOY_DIR` env var still overrides if present.

### What this fixes

Things that used to require operator intervention now Just Work:

- **Moving the install dir.** `mv /tmp/drift-deploy-0.1.39 /elsewhere && rm -rf /elsewhere` is fine — the running stack reads from `/var/lib/drift-cp` and doesn't care where the bundle was.
- **Deleting the install dir.** Operator can `rm -rf` the bundle dir immediately after install.sh prints `✓ install complete`. Future install.sh re-runs from a freshly-extracted bundle land at the same `/var/lib/drift-cp` and pick up the previous values.
- **Recreating drift-agent from the wrong directory.** No more "wait, my `working_dir` label is stale and the LLM modal can't save."

### Migration

On first run of v0.1.39's install.sh, the script detects pre-v0.1.39 installs by inspecting `docker inspect drift-agent`'s `working_dir` label. If that label points to a directory other than `/var/lib/drift-cp` and contains a `.env`, the script:

1. Copies the legacy `.env` + `config/` into `/var/lib/drift-cp/`
2. Overwrites `docker-compose.yml` + templates with the new bundle's versions
3. Re-renders config templates with current prompt answers
4. `docker compose up -d --force-recreate` — necessary on the migration so containers land with the new working_dir + bind paths in one step

After the migration completes, the old install dirs (e.g., `/root/setup/drift-deploy-0.1.37`) are inert — every operational path now points at `/var/lib/drift-cp`. `rm -rf /root/setup/drift-deploy-0.1.37` is safe.

### Operator action

```bash
VERSION=v0.1.39
curl -L "https://github.com/Scope-Creep-Labs/drift/releases/download/${VERSION}/drift-deploy-${VERSION#v}.tar.gz" | tar -xz
cd "drift-deploy-${VERSION#v}"
./install.sh
```

After install completes:

```bash
cd ..
rm -rf "drift-deploy-${VERSION#v}"   # bundle dir is ephemeral, safe to delete
```

For future updates, the same pattern works: extract the new bundle anywhere, run install.sh, delete the bundle dir. The running stack stays at `/var/lib/drift-cp` regardless.

### Backwards compatibility

- Existing `.env` and rendered `config/` from v0.1.37/v0.1.38 installs are preserved through the migration. No prompt re-answers needed.
- API keys, admin password, vmauth credentials, ntfy topic, all survive.
- Docker volumes (Postgres data, VictoriaMetrics data, VictoriaLogs data, etc.) are unaffected.
- `DRIFT_STATE_DIR` env override still works for non-default deployments.
