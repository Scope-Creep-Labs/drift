## v0.1.68

Lock down tunnel subdomains: pasting a tunnel URL into a fresh browser no longer just works. Access now requires a Drift session as the user who minted the tunnel (or an admin).

### Why

v0.1.59 shipped tunnels as URL-as-capability — 128 bits of entropy in the subdomain, "magic link" model. That's defensible in a vacuum, but tunnel URLs leak: browser history, clipboards, screenshots, server access logs, shared by mistake in chat. With a long-lived TTL (4 h default) and an upstream service that the tunnel owner trusts (otherwise why open it), the blast radius of one leaked URL is "anyone who gets it can poke that upstream for 4 hours." That's a worse default than "you must be signed in to Drift first."

### What changed

**`drift-agent/app/deploy/tunnel_proxy.py`** — `TunnelProxyMiddleware` gains an auth check between "look up bridge by subdomain" and "wait for bridge to attach":

- The request's `drift_session` cookie is read out of the ASGI scope.
- The session is resolved to a user. If the user matches the tunnel's `user_id` OR is an admin, the proxy proceeds as before.
- Otherwise: for `Accept: text/html` requests (browsers), 302 to the SPA login at `PUBLIC_URL`. For everything else (curl, scripts), 401 plain text with a pointer at the login URL. WebSocket requests get a 4401 close.
- The 128-bit subdomain token is now defense-in-depth, not the only gate.

**`drift-agent/app/users/routes.py`** — the SPA's `Set-Cookie` on login now passes `domain=<settings.session_cookie_domain>` when configured. Same for `delete_cookie` on logout (browsers only honor an expiring `Set-Cookie` if the original `(name, path, domain)` triple matches). Without a `Domain` attribute the cookie is host-scoped — `dabba.princesamuel.me` only — so it would never reach `tunnel-*.dabba.princesamuel.me` and the new owner check would 401 even when the user is signed in.

**`app/config.py`** — new `SESSION_COOKIE_DOMAIN` setting (default empty). Empty = old host-scoped behavior (production single-host installs that don't use tunnels can leave it alone). Set to `$DOMAIN` = cookie visible to every subdomain (required for tunnels).

**`deploy/install.sh`** — fresh installs now generate `SESSION_COOKIE_DOMAIN=$DOMAIN` automatically (the same `$DOMAIN` derived from PUBLIC_URL that already feeds `TUNNEL_BASE_DOMAIN`).

### Operator action

After Apply:

1. Add `SESSION_COOKIE_DOMAIN=dabba.princesamuel.me` to `/var/lib/drift-cp/.env` (replace with your `$DOMAIN`).
2. Apply again so drift-agent picks up the env change.
3. **Log out + log back in once** — the existing session cookie has no `Domain` attribute so it's invisible to tunnel subdomains. After re-login the new cookie is Domain-scoped and tunnels work transparently for the user who minted them.
4. Existing live tunnels keep working for their owner; you don't have to revoke + remint.

### Migration / breaking changes

A user with an old (no-Domain) session cookie hitting any tunnel subdomain will see the 401/redirect once. After one fresh login the cookie has Domain and they're back to normal flow.

### Security model after this release

| Layer | What it guards against |
|---|---|
| 128-bit subdomain token | URL collisions; broad subdomain probing (`foo.dabba…` doesn't resolve to a real tunnel) |
| Wildcard cert via DNS-01 | No per-subdomain cert issuance leaks via CT logs (every tunnel reuses one cert) |
| Cookie + owner check (NEW) | Stolen URL alone is useless without a valid Drift session as the right user |
| Upstream app's own auth (e.g. Transmission basic auth) | Even with a valid tunnel session, app-level credentials still required for that app's API |

Admins bypass the owner check intentionally (operators should be able to inspect a teammate's tunnel during troubleshooting). To prevent that, revoke the tunnel.

### Cache impact

No system prompt or tool registry changes.
