## v0.1.64

Fix: tunnel HTTP proxy returns 502 with "upstream sent malformed HTTP" on every request.

### Why

`tunnel_proxy.py` used two separate `h11.Connection` instances — one with `role=CLIENT` to send the request to the upstream, and one with `role=SERVER` to parse the response. The second is wrong: the bytes coming back from the upstream are a server's response, but a `role=SERVER` h11 connection treats incoming bytes as a fresh inbound *request*. So h11 saw `HTTP/1.1 301 Moved Permanently` (a status line) and tried to parse it as a method/target/version request line — first token `HTTP/1.1` failed validation → `RemoteProtocolError: illegal request line` → middleware returned 502.

Surfaced the first time we minted a real tunnel: Grafana on the host redirects `/` → `/grafana/` via 301, and that redirect was the first byte sequence the proxy ever tried to parse.

### What changed

`drift-agent/app/deploy/tunnel_proxy.py`:

- Single `h11.Connection(our_role=h11.CLIENT)` now handles BOTH directions. h11's state machine is designed for this — a CLIENT connection sends `Request`/`Data`/`EndOfMessage` and receives `Response`/`Data`/`EndOfMessage` from the upstream on the same instance.

Pre-fix, the proxy 502'd on any upstream that wasn't a perfectly-formed naked HTTP body without a status line — i.e. on essentially everything real.
