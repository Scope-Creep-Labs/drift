## v0.1.60

Fix: existing devices (running v0.1.58 or earlier) can now pick up `tunnel-bridge.py` automatically on the same check-in that auto-updates their agent script — no `install.sh` rerun required.

### Why

v0.1.59 introduced `tunnel-bridge.py` as a new file under `/opt/drift/` on each device, and the self-update path was modeled on `terminal-bridge.py`, which assumed the file was already baked into the image (it's been there for many versions). The `[ -f file ]` precondition meant: if the file was missing, the SHA check was skipped → the file was never fetched → tunnels stayed stuck in `pending` on existing devices forever. The release notes acknowledged that gap by pointing operators at `install.sh`, but that's painful at fleet scale.

### What changed

**`edge-agent/drift-deploy-agent.sh`:**

- The `tunnel-bridge.py` refresh block no longer requires the file to pre-exist. Missing file → treated as `_tbridge_local=""` → mismatch → fetch. First-provision is safe: the download is bearer-authenticated against `BOOTSTRAP_TOKEN` and SHA-verified post-download (same `_new_sha = _target` gate as the update path).
- New `log_info "tunnel-bridge.py provisioned (first-install): → <sha>"` line when the file was missing — distinguishes initial install from later updates in `docker logs`.
- When `pending_tunnels[]` arrives but `/opt/drift/tunnel-bridge.py` is missing (e.g. fetch failed earlier, raced with this tick), the agent now logs a `WARN: N pending tunnel(s) but /opt/drift/tunnel-bridge.py missing — will retry next tick`. Without this, operators saw tunnels frozen in `pending` with nothing in the device logs to diagnose against.

### Impact

After applying v0.1.60 to the CP, all online devices currently on v0.1.58 self-update their agent script on next check-in (≤30s). The new script first-provisions `tunnel-bridge.py` on the same tick. Operators can then open tunnels against those devices without touching `install.sh`.

Drift-agent + drift-frontend images are unchanged in behavior; only `drift-deploy-agent.sh` differs from v0.1.59. The package-release smart-rebuild will still re-tag both `drift-agent:v0.1.60` and `drift-frontend:v0.1.60` because the agent image bundles the script (rebuild = SHA changes).

### Operator action

Apply via Software Updates modal — no Caddy / DNS / `.env` change needed on top of the v0.1.59 setup.

### Migration / breaking changes

None.

### Cache impact

No system prompt or tools-list changes.
