## v0.1.53

Fix v0.1.52's disk-observability collector: paths were `du`'d inside the agent container's mount namespace, returning 4 KB / 12 KB / 20 KB instead of real host bytes.

### Why

Same root cause as v0.1.45's drift-user UID resolution: the edge-deploy-agent container has its own `/var/log`, `/var/cache`, etc. (mostly empty), and `/var/lib/docker/*` isn't bind-mounted in at all. So v0.1.52's collector reported the *container's* view, not the host's:

```
host_path_bytes{path="/var/log"} 4096            # container view (empty)
host_path_bytes{path="/var/cache"} 12288         # container view (empty)
host_path_bytes{path="/var/lib/drift-deploy"} 20480
# /var/lib/docker/* — missing entirely (paths don't exist in container)
```

After the fix, real host data:

```
host_path_bytes{path="/var/lib/docker/overlay2"} 6098984960     # 6.1 GB
host_path_bytes{path="/var/lib/docker/volumes"} 2420228096      # 2.4 GB
host_path_bytes{path="/var/lib/docker/containers"} 138424320    # 138 MB
host_path_bytes{path="/var/log"} 4375302144                     # 4.4 GB
host_path_bytes{path="/var/cache"} 476782592                    # 477 MB
host_path_bytes{path="/var/lib/drift-deploy"} 20480
```

### What changed

`du` and `test -d` calls in `collect_disk_observability` now run inside the host's mount namespace via `nsenter -m -t 1`. Same trick as the drift-user UID lookup — PID 1 is the host's init (reachable because the agent runs `--pid host`), so its mount NS is the host root.

`docker_disk_bytes` and friends were already correct in v0.1.52 — `docker system df` goes through the docker socket which doesn't care about mount namespaces.

### Operator action

Apply via Software Updates modal. Image-only — drift-agent's image embeds the new bash agent. Edge devices self-update to `AGENT_VERSION=0.13.3` on the next check-in tick, first correct disk-observability scrape lands within an hour after.

