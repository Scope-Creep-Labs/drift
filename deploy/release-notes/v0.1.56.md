## v0.1.56

Fix: fleet-wide investigations weren't consulting operator filters, so noise like cAdvisor's `product_name` error on pi-riffpod-001 still showed up in "how's the fleet doing?" summaries even with a matching filter saved.

### Why

Two prompt-level triggers both said "scoped to a specific device, group, or container" as a precondition for calling `list_relevant_filters`:

- `SYSTEM_PROMPT` section 6 (operator filters)
- `list_relevant_filters` tool description

Fleet-wide questions ("how's the fleet doing?", "any errors?") aren't scoped — so the agent treated them as exempt and skipped the filter lookup entirely. The filter would have *matched* if asked — v0.1.48's compatible-scope semantics already returns `device=pi-riffpod-001` filters on a fleet-wide query — but the agent never reached that code path.

### What changed

Both triggers broadened to "ANY investigation that may surface errors / alerts / noisy logs — INCLUDING fleet-wide". The system prompt now explicitly tells the agent to call `list_relevant_filters({})` (empty scope) for fleet-wide questions, returning every filter visible to the operator. The tool description mirrors the same instruction.

### Operator action

Apply via Software Updates modal. Image-only — `drift-frontend` unchanged.

### Cache impact

One cache miss on first turn after upgrade — both SYSTEM_PROMPT and the tools list serialize differently. Steady-state caching resumes from the second call.
