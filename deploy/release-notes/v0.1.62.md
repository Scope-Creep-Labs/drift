## v0.1.62

Three fixes for the tunnel feature — surfaced end-to-end during the first real attempt to mint a tunnel against a managed device. The feature was 95% wired; these are the last 5%.

### What changed

**`drift-agent/app/deploy/tunnel.py`:**

- `_gen_subdomain_token` now uses `secrets.token_hex(16)` (32 lowercase hex chars, 128 bits) instead of `token_urlsafe(16)` (mixed-case base64url). Caddy lowercases the SNI hostname **before** calling the on-demand-TLS ask hook, so a mixed-case token round-tripped through Caddy as a different string than what was stored in `tunnel_sessions`. Ask returned 404 → cert refused → browser saw a TLS internal-error. Symptom: tunnel status flipped to `active`, but the URL never opened — `curl: (35) ... tlsv1 alert internal error`.

**`nginx.conf`** (drift-frontend's config):

- Regex that adds `Upgrade`/`Connection: upgrade` headers for WebSocket endpoints now matches `/api/deploy/agent/tunnel/ws/...` alongside the existing `/api/deploy/agent/terminal/ws/...`. Without this, the edge agent's tunnel-bridge WS request fell through to the generic `/api/` block (no WS upgrade) → drift-agent's FastAPI websocket route returned 404. Symptom: `tunnel-bridge: server rejected WebSocket connection: HTTP 404` in the edge-agent docker logs, tunnel stuck in `pending` forever.

**`deploy/install.sh`:**

- The Caddy snippet printed to operators in external-proxy mode had the wrong ask-hook path — `/api/internal/tunnel/check` (which doesn't exist) instead of `/api/deploy/internal/tunnel/check` (the actual route, prefixed by tunnel router's `/api/deploy` mount). On a fresh install where the operator pasted the snippet verbatim, Caddy would call a 404 endpoint and never issue any tunnel cert. Existing installs that pasted the snippet need to update their Caddyfile's `on_demand_tls.ask` URL to include `/deploy/` — five-character fix, no restart needed beyond `caddy reload`.

### Why

The mint endpoint + DB row + edge-bridge spawn all worked first try. The three blockers were all "wires crossed" details: case sensitivity in the cert path, missing nginx WS upgrade for the new tunnel WS endpoint, and a copy-paste typo in the install-time Caddy snippet. Independent failures, one per layer (auth → routing → cert), each silent in its own way.

### Operator action

Apply via Software Updates modal. The drift-agent + drift-frontend images both change (tunnel.py + nginx.conf). After apply:

1. **If you've already pasted the Caddy snippet from a previous install run**, edit your host's Caddyfile and change `ask http://localhost:10001/api/internal/tunnel/check` → `ask http://localhost:10001/api/deploy/internal/tunnel/check`, then `caddy reload`. Fresh installs get the corrected snippet from this version's `install.sh`.

2. **Existing mixed-case tunnel rows** in `tunnel_sessions` (from pre-v0.1.62 tries) will keep working for in-process subdomain proxying but will fail at the cert-issuance step on a fresh browser hit. Easiest: revoke them via the Tunnel modal and mint fresh ones — new tokens are lowercase hex by construction.

### Migration / breaking changes

None. The tunnel subdomain shape is unchanged (`tunnel-<token>.<base>`); only the token character set narrowed.

### Cache impact

No system prompt or tool registry changes.
