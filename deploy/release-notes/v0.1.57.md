## v0.1.57

New `DEMO_MODE` env flag — when on, the CP refuses operator-side mutations that would either compromise a shared demo account's experience or corrupt the simulator's fixed device fleet, enforces a per-session investigation-turn cap, and surfaces a banner in the UI. Default off; existing deployments are unaffected.

### Why

Pairs with the new [drift-demo](https://github.com/Scope-Creep-Labs/drift-demo) simulator. Demo visitors log in with a shared `demo` account and need to be able to play with chat investigations, filters, deploys, and alerts — without being able to change the admin password, rotate the LLM key, commission/delete the simulator's 100 devices, or burn an unbounded LLM budget.

### What's new

**Config (`app/config.py`):**

- `DEMO_MODE` (default `false`)
- `DEMO_MAX_TURNS_PER_SESSION` (default `10`) — per-session investigation cap on `/api/investigate`
- `DEMO_BANNER_MESSAGE` — text the SPA banner shows

**Backend route guards (return 403 with a clear `"blocked in DEMO_MODE …"` message):**

- `POST /api/auth/users` · `PATCH /api/auth/users/{username}` · `DELETE /api/auth/users/{username}`
- `POST /api/deploy/devices` (commission) · `DELETE /api/deploy/devices/{name}`
- `PUT /api/deploy/registry-creds` · `DELETE /api/deploy/registry-creds/{registry}`
- `PUT /api/admin/llm-settings`
- `POST /api/admin/updates/apply`

Deploy / app-revision / filter / alert-management endpoints stay open in demo — the simulator pretends to apply revisions so the demo user feels the full flow.

**Chat-tool guards (refuse with an error dict the agent relays):**

- `commission_device`, `delete_device`
- `create_user`, `delete_user`, `reset_user_password`

**Per-session turn cap on `/api/investigate`:**

In-memory counter keyed by session_id (the auth cookie). When a session exceeds `DEMO_MAX_TURNS_PER_SESSION`, the endpoint returns `429` with `{error: "demo_turn_cap_exceeded", cap, remaining: 0}`. Sign out + back in for a fresh budget. The map evicts entries older than 7 days when it grows past 5000 sessions.

**New public endpoint `GET /api/auth/demo/status`** — no auth required, returns `{demo_mode, banner_message}` so the login page can render a demo label before the visitor enters credentials.

**`/me`, `/login`, `/users` responses gain `demo_mode` + `demo_banner_message`** so the SPA has the flag on every auth response. No extra round-trip on page load.

**Frontend:**

- `DemoBanner` — sticky info bar at the top of the viewport when `me.demo_mode` is true. Renders nothing otherwise (zero overhead on non-demo deploys).
- Sidebar hides the LLM settings, Software updates, and Registry credentials icons in demo mode (the backend would refuse them anyway). Change-password + Sign-out icons stay visible since those are per-user operations.

### Operator action

Apply via Software Updates modal. No behavior change unless `DEMO_MODE` is set in the env. To turn the demo on:

```bash
# .env or docker-compose env
DEMO_MODE=true
DEMO_MAX_TURNS_PER_SESSION=10
DEMO_BANNER_MESSAGE="Demo mode — actions visible to other operators. Resets nightly."
```

### Migration / breaking changes

None. Existing deployments without `DEMO_MODE` set behave identically to v0.1.56.

### Cache impact

No prompt or tools-list changes — system prompt and the agent's tool registry are unchanged. Cache prefix stays byte-stable across the upgrade.
