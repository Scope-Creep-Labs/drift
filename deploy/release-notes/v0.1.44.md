## v0.1.44

Image-only release. Two new `DRIFT_*` env vars exposed to deployed apps, killing the most common shape of per-device PUID/PGID + bind-path setup.

### What's new

- **`DRIFT_USER_UID` + `DRIFT_USER_GID`** — UID/GID of the host's `drift` user (the account `install.sh` provisions on every device for web-terminal sessions). LinuxServer.io-style images and anything else that takes `PUID`/`PGID` can now just do:

  ```yaml
  environment:
    - PUID=${DRIFT_USER_UID}
    - PGID=${DRIFT_USER_GID}
  ```

  Works across distros — install.sh's `useradd` lands at different UIDs on Alpine vs Debian vs Synology DSM, but the `drift` user always exists and the bash agent resolves it once at startup via `id -u drift` / `id -g drift`. Falls back to 1000:1000 if the user is somehow missing (defensive — install.sh failures, DSM edge cases).

- **`DRIFT_APP_STATE_DIR`** — per-app persistent directory at `/var/lib/drift-deploy/apps/<app>/state/`. Lives alongside the per-revision dirs the agent creates on each apply, but survives revision bumps. Auto-`mkdir`ed and `chown`ed to `drift:drift` on first apply. Mount paths under it (instead of `./`) for state that the container itself writes:

  ```yaml
  volumes:
    - ${DRIFT_APP_STATE_DIR}/config:/config
    - ${DRIFT_APP_STATE_DIR}/downloads:/downloads
  ```

  Before this you had to pick between (a) named docker volumes, which dump everything into `/var/lib/docker/volumes/`, or (b) an absolute host path, which means operator-once `mkdir + chown` per device. The new var gives you a third option that needs zero operator setup.

### Why this matters

Drift's existing `./` relative-path convention only works for files the bundle SHIPS (e.g. reporter's `./prometheus.yml`). It doesn't work for state the container itself writes, because each revision bump creates a new per-rev UUID directory and the bundle ships no initial state. Container generates fresh config in v1's dir; v2 push starts with an empty dir; state lost.

`DRIFT_APP_STATE_DIR` fixes that for the small-state case (configs, sqlite DBs, settings). Bulk data (multi-GB Plex/Transmission downloads) can still use absolute paths when you want it on a specific disk — and `chown drift:drift` is now the cross-distro standard recipe instead of "figure out which UID your operator's local user has".

### Example: Transmission

```yaml
services:
  transmission:
    image: lscr.io/linuxserver/transmission:latest
    container_name: transmission
    environment:
      - PUID=${DRIFT_USER_UID}
      - PGID=${DRIFT_USER_GID}
      - TZ=America/Los_Angeles
      - USER=transmission
      - PASS=${PASSWORD}
    volumes:
      - ${DRIFT_APP_STATE_DIR}/config:/config
      - ${DRIFT_APP_STATE_DIR}/downloads:/downloads
      - ${DRIFT_APP_STATE_DIR}/watch:/watch
    restart: unless-stopped
```

Zero operator setup, state persists across revision bumps, downloads live on the same partition as the rest of the agent state (fine for flat-filesystem hosts like a Pi; switch `/downloads` to an absolute path if you want it elsewhere).

### Operator action

Image-only release. drift-agent on the CP picks it up via the Software Updates modal (one click). Each edge device's bash agent self-updates on its next check-in tick (~30s per device — `AGENT_VERSION` bumps from 0.12.0 to 0.13.0). After that the new vars are available in every subsequent `apply_revision` call. Existing apps are unaffected until you author a new revision that uses them.
