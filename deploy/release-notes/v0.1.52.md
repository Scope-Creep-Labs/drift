## v0.1.52

Disk-usage attribution: the agent can now answer "where did the space go on host X?" from metrics alone.

### Why

Caught when investigating debian-8gb-ash-1 at 91% root-fs full. `container_fs_usage_bytes` (cAdvisor's per-container writable-layer metric) totalled 85 KB across 23 running containers — accurate but useless for the actual question. The real consumers (38 GB of stale image layers, 13 GB of build cache, 6 GB of dangling volumes, 36 GB of dev trees in `/root`) were invisible to PromQL.

### What's new

The edge-deploy-agent now runs an **hourly disk-observability collector** that writes to a separate textfile (`drift_disk_observability.prom`) scraped by node-exporter. Three new metric families:

- **`host_path_bytes{path, host}`** — allocated bytes (`du -B1 -sx`) for a curated set of paths the operator usually cares about:
  - `/var/lib/docker/overlay2` (image + writable layers)
  - `/var/lib/docker/volumes` (named + anonymous volumes)
  - `/var/lib/docker/containers` (json.log files)
  - `/var/lib/docker/builder` (build cache, when present)
  - `/var/log`, `/var/cache`, `/var/lib/drift-deploy`

- **`docker_disk_bytes{kind, host}`** — total bytes per docker artifact category, parsed from `docker system df`. Kinds: `images`, `containers`, `local_volumes`, `build_cache`.

- **`docker_disk_reclaimable_bytes{kind, host}`** — the reclaimable subset per category. This is the killer — `docker_disk_reclaimable_bytes{kind="images"}` answers "how much would `docker image prune -a` free?" without needing a shell.

Plus `docker_disk_artifacts{kind, state, host}` (state ∈ total / active) and `host_disk_observability_last_run_timestamp_seconds{host}` so the agent can tell whether the data is fresh.

### Why hourly, not per-tick

`du` on `/var/lib/docker/overlay2` can take minutes on slow USB storage (Pi, Synology). Running every 30s would thrash slow devices for data that trends slowly. Real-time disk-fill alerts already work off `node_filesystem_avail_bytes` at 30s resolution; this new collector is for **attribution** — "where did the space go?" — not alerting.

The collector is:
- Wrapped in `nice -n 19 ionice -c 3` so it doesn't steal I/O from other processes.
- Guarded by `flock -n` — if a prior run is still going (slow storage), the next hour's tick skips cleanly instead of piling up.
- Backgrounded with fd 9 closed so it doesn't hold the main reconcile lock for the duration.
- Marker file written BEFORE the run, so a stuck collector doesn't trigger back-to-back attempts.

`DISK_OBS_INTERVAL_SECONDS` env override is wired in case an operator wants tighter or looser cadence.

### Security

The collector runs **inside the existing edge-deploy-agent container** — already root, already has `/var/run/docker.sock` mounted, already writes to the textfile_collector dir. **Zero new attack surface.** No new sidecar container, no new bind mounts, no new docker-socket exposure.

### Agent prompt

`SYSTEM_PROMPT` updated to document the new metric families and the cAdvisor caveat (`container_fs_usage_bytes` uses label `name`, not `container_name`, AND only covers writable layers, so it's the wrong tool for disk-pressure investigations).

### Operator action

Apply via Software Updates modal. Image-only — drift-agent's image embeds the new bash agent script, edge devices self-update to `AGENT_VERSION=0.13.2` on the next check-in tick after the CP picks up the new image. First disk-observability scrape lands within an hour of the device self-update.

### Cache impact

One cache miss on the next investigation after upgrade — `SYSTEM_PROMPT` gained the disk-observability section. Steady-state caching resumes from the second call.
