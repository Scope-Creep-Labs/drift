## v0.1.65

Fix: tunnel HTTP proxy 500s on any POST/PUT with a body — "Too much data for declared Content-Length" in drift-agent logs.

### Why

`_build_request_headers` was stripping `Content-Length` from the request headers I rebuild for the upstream. The comment claimed h11 would re-compute it from outgoing `Data()` events. It doesn't — h11 needs the framing header set on the `Request` object before you call `.send()`, otherwise the connection is in "no-body" mode and the first `Data()` raises `LocalProtocolError: Too much data for declared Content-Length`. GET worked (no body); POST with body failed reliably. Caught on the first real attempt to use a debug UI through the tunnel (Transmission's `/transmission/rpc`).

### What changed

`drift-agent/app/deploy/tunnel_proxy.py`:

- Removed `b"content-length"` from the hop-by-hop strip list in `_build_request_headers`. The browser's framing (Content-Length OR Transfer-Encoding) now passes through to h11, which frames the upstream request correctly.
- `b"transfer-encoding"` also removed from strip list — same reason. `Transfer-Encoding: chunked` requests are now framed as chunked on the way to the upstream too.
- Belt-and-suspenders: if both Content-Length and Transfer-Encoding: chunked are present on the inbound request (which RFC 7230 forbids but a misbehaving client could send), Content-Length is dropped so h11 falls back to chunked framing instead of erroring.
