## v0.1.47

The investigation agent can now **learn from the operator across sessions**. When the operator says "ignore that cadvisor error on pi-riffpod-001 in future reports," the agent persists a noise-suppression rule and applies it on the next investigation that touches the same scope.

### What's new

Three new agent tools backed by a single new table (`operator_filters`):

- **`remember_filter(pattern, scope, reason)`** — persist a noise rule. The agent calls this when the operator says "ignore X" / "treat Y as noise" / "make a note to skip Z next time". `pattern` is a case-insensitive substring; `scope` is a sparse dict (any combination of `device`, `container`, `group`, `signal`); `reason` is the operator's WHY, stored verbatim so the rule's relevance can be judged later.

- **`list_relevant_filters(device?, container?, group?, signal?)`** — return filters that apply to the current investigation scope. The agent calls this at the **start** of any device/group/container-scoped investigation, BEFORE summarizing errors or alerts. A filter matches when every key it pins equals the request's value (empty filter scope = wildcard / user-wide). Bumps `apply_count` + `last_applied_at` per call for usefulness signal.

- **`forget_filter(filter_id)`** — remove a rule. The agent calls this when the operator says "stop ignoring X" / "remove that filter".

The system prompt instructs the agent to suppress matches from its narrative and append a small footer like `_suppressed N lines matching M operator filters_` so over-silencing is visible.

### Design notes (load-bearing — read before extending)

- **Per-user, not per-fleet.** Filters live on `users.id` with `ON DELETE CASCADE`. There is no fleet-wide promote tool in v1 — promoting a personal rule to a shared one would mean copying for each member of a user-group. Acceptable for the single-operator / small-team setups Drift targets today.
- **Case-insensitive SUBSTRING matching only.** No regex, no wildcards. Keeps the surface minimal, avoids ReDoS, and means matching happens in the agent's own summary code path against any string it'd otherwise emit.
- **Scope is JSONB**, not flat columns. Adding new narrowing dimensions later (e.g. `image`, `severity`) is a tool-schema change with no migration.
- **Prompt cache stays byte-stable.** Filters don't go into the system prompt — they arrive via tool results, so the prefix stays cacheable across calls. Same convention as alerts / deploy state.
- **Idempotency.** Saving the same `pattern + scope` twice creates two rows. Intentional — the operator may want different `reason`s on the same span, and the agent can suggest consolidation if the count gets out of hand.

### Schema

```sql
CREATE TABLE operator_filters (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    pattern TEXT NOT NULL,
    scope JSONB NOT NULL DEFAULT '{}'::jsonb,
    reason TEXT NOT NULL DEFAULT '',
    created_at TIMESTAMPTZ NOT NULL,
    last_applied_at TIMESTAMPTZ NULL,
    apply_count INT NOT NULL DEFAULT 0
);
CREATE INDEX ix_operator_filters_user_id ON operator_filters(user_id);
```

### Operator action

Apply via Software Updates modal in the UI (one-click). Image-only — no install.sh re-run, no edge agent change. Alembic migration `0012_operator_filters` runs on container startup.

### Future work (not in this release)

None. New table only; no impact on existing flows.
