## v0.1.66

New **Telegram** integration — two sibling features against the same bot:

1. **Chat with Drift's agent over Telegram.** Operators bind their Drift account to a chat with a one-time `/link <code>` and then just message the bot ("which devices look unhealthy?", "CPU on debian-8gb-ash-1 for the last hour", etc.). Each chat keeps per-conversation memory by sliding into Drift's existing `_session_history` under `investigation_id="telegram:<chat_id>"`, so multi-turn questions work the same as in the SPA.
2. **Alertmanager → Telegram bridge.** A webhook receiver on drift-agent translates each firing/resolving alert into a Telegram message and fans it out to every chat in `TELEGRAM_ALERT_CHATS`. Sibling to (not replacement for) the existing `alertmanager-ntfy` push path — operators who prefer ntfy keep it.

Bootstrapped from the FamilAI agent-starter's Telegram implementation (`familai/agent/app/telegram_*.py`); the three generic layers (API wrapper, /link flow, long-poll loop) port mostly verbatim, with the user-resolution layer adapted to Drift's pg + SQLAlchemy.

### What's new

**Backend:**

- `app/telegram/api.py` — async Telegram Bot API client (httpx). `get_updates` for long-poll, `send_message` (Markdown → Telegram HTML with plain-text fallback), `send_chat_action` ("typing…"), `get_me` + `deep_link` for `/link` QR/button flows.
- `app/telegram/links.py` — `issue_link_code`, `redeem_code`, `user_for_chat`, `list_chats_for_user`, `unlink_chat`. Operates on two new tables.
- `app/telegram/bot.py` — the long-poll loop. Intercepts `/start`, `/link`, `/reset`; routes everything else to `run_agent` with `investigation_id=f"telegram:{chat_id}"`. SSE events from the agent are parsed back and collapsed into a single chat reply (narrative + markdown blocks rendered as-is, tables shown inline up to 8 rows, charts/timelines summarized as `[chart omitted]`).
- `app/telegram/alerts.py` — Alertmanager webhook → formatted Telegram message per alert. Severity-aware icon (🚨/⚠️/ℹ️/✅) plus a footer of useful labels (`host=…`, `instance=…`, `severity=…`).
- `app/telegram/routes.py` — operator endpoints (`POST /api/telegram/link/code`, `GET /api/telegram/chats`, `DELETE /api/telegram/chats/{id}`) and the Alertmanager webhook (`POST /api/telegram/alertmanager/webhook?secret=…`).

**Database (migration `0016_telegram`):**

- `telegram_link_codes` — short-lived `/link` codes (6 chars, default 10 min TTL).
- `telegram_chats` — resolved chat ↔ user bindings; lookup direction is `chat_id → user_id`.

**Config (`app/config.py`):**

- `TELEGRAM_BOT_TOKEN` (default `""`) — the BotFather token. Empty disables the whole feature; everything else (loop, routes, webhook) becomes a no-op / 503.
- `TELEGRAM_POLL_TIMEOUT` (default `30`) — long-poll wait per `getUpdates`.
- `TELEGRAM_LINK_CODE_TTL_MIN` (default `10`).
- `TELEGRAM_ALERT_CHATS` (default `""`) — comma-separated chat IDs the Alertmanager bridge sends to. Empty = chat side still works, alert side stays dormant.
- `TELEGRAM_WEBHOOK_SECRET` (default `""`) — shared secret in `?secret=…` for the Alertmanager webhook. Empty = webhook 404s even when the bot is otherwise live.

**Lifecycle:**

- Bot loop starts in `main.py`'s `startup` only if `TELEGRAM_BOT_TOKEN` is set; cancelled cleanly on shutdown.

### Operator action

To enable both surfaces, after applying:

1. **Create a bot** via BotFather (`/newbot` in the BotFather chat), grab the token.
2. **In `/var/lib/drift-cp/.env`:**
   ```
   TELEGRAM_BOT_TOKEN=<token from botfather>
   TELEGRAM_WEBHOOK_SECRET=<openssl rand -base64 24>
   TELEGRAM_ALERT_CHATS=<your chat id, comma-separated for multiple>
   ```
3. **Apply via Software Updates** (the bot loop starts on next startup).
4. **Bind your account:** In the Drift admin, `curl -X POST -b "drift_session=…" https://dabba.princesamuel.me/drift/api/telegram/link/code` returns a 6-char code + a `https://t.me/<bot>?start=<code>` deep link. Tap it (or type `/link <code>` in the bot). The bot confirms with ✅ Linked.
5. **Wire Alertmanager** by adding a receiver:
   ```yaml
   receivers:
     - name: telegram
       webhook_configs:
         - url: https://dabba.princesamuel.me/drift/api/telegram/alertmanager/webhook?secret=<TELEGRAM_WEBHOOK_SECRET>
           send_resolved: true
   ```
   …and routing the firing rules you want delivered through that receiver (alongside or instead of ntfy).

Step 4's SPA modal ("Link Telegram") will ship in a follow-up release; the API works today.

### Migration / breaking changes

`alembic 0016_telegram` adds two tables. No existing rows change.

### Known limitations (ship in 0.1.67+)

- **No frontend modal yet** for issuing `/link` codes — operators use the API directly (or the Drift admin's HTTPie-style flow).
- **No photo / document handling** — text-only chat for now (FamilAI's ingest-attachment path is FamilAI-specific anyway).
- **Conversation memory is in-memory** (`_session_history` is a dict on the drift-agent process). A CP restart drops Telegram conversation context the same way it drops SPA investigations' in-memory state. Cookie-equivalent durability isn't a problem we're solving in this release.
- **DEMO_MODE turn cap doesn't apply** — Telegram-driven turns don't go through `/api/investigate`, so they bypass the demo per-session cap. If you turn DEMO_MODE on AND give the bot to demo visitors, that's a budget hole worth closing in a follow-up.

### Cache impact

No system prompt or tool registry changes. The Telegram bot reuses the existing `run_agent` entrypoint — same cache prefix.
