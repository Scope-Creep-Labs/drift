## v0.1.48

Fixes v0.1.47's operator-filter matching: filters with extra scope keys (e.g. `{device, signal}`) were silently excluded from broader queries (e.g. `device`-only).

### Why

Caught on first real use. Operator saved a filter:

```json
{"pattern": "product_name: no such file ...", "scope": {"device": "pi-riffpod-001", "signal": "log"}, "reason": "..."}
```

Then asked "any errors on the pi?". The agent dutifully called `list_relevant_filters(device="pi-riffpod-001")` — but the saved filter pinned `signal=log` too, and the old matcher required every filter key to be present in the request. `signal` absent from request → mismatch → 0 filters returned → footer reported `suppressed 0 lines matching 0 operator filters`.

### What changed

**`_scope_matches` is now COMPATIBLE-scope semantics, not strict-containment.** A filter matches the request when no key pinned by **both** has a conflicting value. Keys the filter pins that the request omits are NOT a disqualifier — the agent applies those constraints per-line against each log/error's actual source.

Truth table:

| Filter scope | Request scope | Match? | Why |
|---|---|---|---|
| `{device: pi, signal: log}` | `{device: pi}` | ✓ | request omits signal — filter still returned, agent checks signal per line |
| `{device: pi, signal: log}` | `{device: pi, signal: alert}` | ✗ | signal conflicts |
| `{device: pi, container: x}` | `{device: pi}` | ✓ | filter narrower, agent applies container check per line |
| `{device: pi}` | `{device: pi, container: x}` | ✓ | filter broader |
| `{}` | `{device: pi}` | ✓ | user-wide filter |
| `{device: pi}` | `{device: other}` | ✗ | device conflicts |

**System prompt section 6 (operator filters) clarified** to instruct the agent on per-line application: drop a line iff (a) the filter's pattern matches the line text AND (b) the line's actual source is consistent with every key the filter pins.

### Cache impact

The system prompt changes between v0.1.47 and v0.1.48, so the first investigation after upgrade pays one cache miss against `system + tools` prefix. Steady-state caching resumes from the second call onward.

### Operator action

Apply via Software Updates modal. Image-only. Existing filters from v0.1.47 work as-is — no schema change.
