## v0.1.69

Per-volume disk attribution. The hourly disk-observability collector on each device now emits one `docker_volume_bytes{name="…"}` series per named Docker volume in addition to the existing `host_path_bytes{path="/var/lib/docker/volumes"}` lump total.

### Why

The agent could tell you "your local-volumes total is 6.6 GB" but couldn't tell you which volume in that pool was responsible. So "how much does the ente-postgres-1 volume use?" had no answer — only an indirect guess via the lump number. With the new metric the agent picks the right series straight out of `list_metrics` and the answer is a `topk(N, sum by (name) (docker_volume_bytes{host="…"}))` away.

### What changed

**`edge-agent/drift-deploy-agent.sh`:**

- `collect_disk_observability` now loops `/var/lib/docker/volumes/*/_data` via `nsenter -m -t 1` (same host-namespace trick the existing host_path_bytes section uses) and `du -B1 -sx` each volume's data tree individually. Output: one `docker_volume_bytes{name="<volume_name>"}` gauge per volume, with the same nice + ionice priority so a 100-volume host doesn't IO-thrash the device.
- Skips the internal `backingFsBlockDev` marker file and any half-created volumes whose `_data` directory is missing.
- `AGENT_VERSION` bumped `0.14.0` → `0.14.1` (script changed; per the agent-version-bump rule).

### Operator action

Apply via Software Updates modal. Each device's `drift-deploy-agent.sh` self-updates within one check-in cycle (≤30 s). The new `docker_volume_bytes` series start landing within one collector tick after that — collector runs hourly by default, so first sample arrives at most ~1 h after the script update. To force an immediate refresh on a specific device, the operator can open a terminal and `rm -f /var/lib/drift-deploy/state/.disk-obs-last-run` (the marker file) so the next reconcile tick re-runs the collector.

### Migration / breaking changes

None — the existing `host_path_bytes` and `docker_disk_bytes` series remain untouched; this only adds the new metric.

### Cache impact

No system prompt or tool registry changes. The agent discovers `docker_volume_bytes` via `list_metrics` the next time the operator asks a volume-related question.
